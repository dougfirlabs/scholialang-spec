# Scholia v0.2 extension proposal - `<Cost>` and `<Budget>` atoms

**Status:** accepted for v0.2
**Target version:** Scholia v0.2
**Impact:** adds two meta atoms, no new validity rule.

## 1. Motivation

The GCN experiment needs token-efficiency and budget-conformance metrics. v0.1 can describe work performed, but cannot record resource caps or measured consumption in a machine-readable way.

## 2. Proposed atoms

```xml
<Budget id="Budget_01" for="Goal_01" tokens="50000" actions="20" wall_clock_ms="600000" />
```

`<Budget/>` declares an intended cap before execution. It usually points at a `<Goal>` or `<Step>`.

```xml
<Cost id="Cost_01" for="Action_03" tokens="1820" wall_clock_ms="4300" dollars="0.12" />
```

`<Cost/>` records measured consumption after an action or other accountable atom.

## 3. Numeric discipline

`tokens`, `actions`, and `wall_clock_ms` are integers. `dollars` is parsed as a JSON/YAML number using Python `float`. v0.2 does not add money arithmetic; it records values. Future enforcement can move to Decimal if sub-cent accounting becomes required.

## 4. Validity

Both atoms use `for` as a normal reference field and must resolve under the existing reference-completeness rule. v0.2 intentionally does not enforce Cost <= Budget; that is a likely v0.3 rule once policy semantics are ready.

## 5. Backward compatibility

Existing traces remain valid with no Cost or Budget atoms.

## 6. Alternatives considered

1. Put cost fields on `<Action>`. Rejected because not every measured cost is an Action, and budget caps often attach to Goals or Steps.
2. Store all values as strings. Rejected because downstream metrics should not need to parse prose.
