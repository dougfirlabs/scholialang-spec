# Scholia v0.2 extension proposal — `<Goal>` atom

**Status:** proposed (v0.2 candidate)
**Author:** Claude Opus 4.7 (dogfooding v0.1 on 2026-04-20)
**Target version:** Scholia v0.2
**Impact:** adds one new atom to the "meta" category, adds one new validity rule, no breaking changes to v0.1 traces.

---

## 1. Motivation

Every Scholia trace written so far — the merge-readiness runbook, today's prose→Scholia dogfooding exercise, the hypothetical examples in NOTATION_REFERENCE.md — opens with an implicit statement of *what the agent is trying to accomplish*. Currently that statement lives inside a `<Thinking>` block as prose:

```xml
<Thinking>
  "Three feature branches exist. Need to answer: do they still merge
  cleanly, and does the merge re-introduce anything that main recently
  purged (namely, the egg-info gunk from commit 7bc963bc)?"
</Thinking>
```

This is a **goal**, not a deliberation. Goals and deliberations have different properties and different verification obligations. Conflating them costs three things:

1. **Monitor evaluation cannot run.** The GCN experiment plan (`.ralph/plans/scholia_gcn_experiment_plan.md` §6b) defines "I/O conformance" as *"Does the final Action satisfy the declared Constraints?"* — but there is no atom in which to declare the **task** itself. A Monitor cannot mechanically verify *"did Subject pursue the stated goal?"* because the goal has no mechanical representation.
2. **Regulatory answer is weak.** When "what was this AI trying to do?" becomes a legally required question, pointing at a `<Thinking>` block is a worse answer than pointing at a `<Goal>` atom.
3. **Multi-agent handoffs leak context.** When Agent_A hands off to Agent_B, Agent_B needs the goal, not the deliberation history. Today's `<Handoff>` atom references a work package but does not carry the goal explicitly.

## 2. Proposed atom

### 2a. Signature

```xml
<Goal id="Goal_01" scope="trace|step" priority="required|optional">
  "Plain-language statement of what the agent is trying to accomplish."

  success_criteria = LIST:
    - "Criterion 1 (optionally REFER: an existing Constraint atom)"
    - "Criterion 2"

  related_constraints = LIST:[REFER:C_01, REFER:C_02]

  deadline = "ISO-8601 timestamp" (optional)

  failure_modes = LIST:
    - "What counts as this goal having failed, not merely deferred"
</Goal>
```

### 2b. Semantics

- **`scope="trace"`** (default): the goal applies to the entire trace. Exactly one trace-scoped Goal per trace is the common case; multiple trace-scoped goals are allowed for genuinely multi-goal tasks (rare).
- **`scope="step"`**: the goal applies only to a specific `<Step>`. Used when a step pursues a sub-goal that isn't the whole trace's purpose.
- **`priority`**: `required` means the trace is considered failed if the goal is unmet; `optional` means the goal is best-effort.
- **`success_criteria`**: list of mechanical checks a Monitor can evaluate against the final state. May reference existing `<Constraint>` atoms.
- **`related_constraints`**: explicit links to governing Constraints in scope. Lets validators cross-check "are constraints for this goal being respected?"
- **`deadline`**: optional ISO-8601 timestamp. If present, a late `<Finding>` relative to this deadline is a soft failure signal even if criteria are met.
- **`failure_modes`**: enumerated conditions that should cause the trace to `<Retract>` prior Findings and escalate. Distinct from unmet criteria — criteria are *acceptance*, failure modes are *disqualification*.

### 2c. Category

`<Goal>` joins the **meta** category alongside `<Constraint>` and `<Confidence>`. Rationale: Goals govern the trace the way Constraints govern Actions — they are scope-setters, not in-flight reasoning. Keeping them in the meta tier preserves the six-category structure.

## 3. Proposed validity rule 8

> **Rule 8 (Goal declaration):** A trace with `priority="required"` Goals MUST contain at least one `<Finding>` per required Goal that declares the goal's status (`met` | `unmet` | `partially_met` + explanation). A trace with no declared Goals is valid only if it is explicitly marked as a research-mode trace via `<Meta:research-mode/>` (self-closing research marker — also v0.2).

This rule converts the current implicit "what was this trace about?" into a mechanically checkable property.

### Why rule 8 is additive, not breaking

Existing v0.1 traces have no `<Goal>` atoms. Under v0.2's rule 8, they trivially satisfy the rule because they have no `priority="required"` Goals. v0.1 traces become valid v0.2 traces with no modification. **No migration required.**

Traces written in v0.2 that wish to declare a goal gain the enforcement; traces that don't, don't. Graceful path.

## 4. Worked examples

### 4a. Canonical single-goal trace

```xml
<Goal id="Goal_01" scope="trace" priority="required">
  "Determine whether the three feature branches merge cleanly into
  current main, without re-introducing the egg-info drift recently
  purged by commit 7bc963bc."

  success_criteria = LIST:
    - "Per-branch merge-tree output contains no CONFLICT lines"
    - "Post-merge trees contain no src/example.egg-info files"
    - "Post-merge .gitignore retains *.egg-info/ and venv/ rules"

  failure_modes = LIST:
    - "Any merge produces modify/delete or content CONFLICT"
    - "Any merged tree resurrects egg-info files"
</Goal>

<Step id="Step_01" name="Fetch + confirm main HEAD"> ... </Step>
...
<Finding id="Finding_final" for_goal="Goal_01" status="met">
  "All three branches merge-clean. egg-info stays absent.
  gitignore rules preserved. See REFER:Step_06, REFER:Step_07."
</Finding>
```

### 4b. Goal with deadline

```xml
<Goal id="Goal_02" scope="trace" priority="required">
  "Ship encrypted binary to Union AI."
  success_criteria = LIST:
    - "Tarball hash communicated out-of-band"
    - "IP counsel sign-off on LICENSE"
  deadline = "2026-04-28T17:00:00-07:00"
</Goal>
```

A `<Finding>` produced after the deadline but satisfying the criteria would be `status="met_late"` — Monitor can distinguish "correctly done" from "correctly done in time."

### 4c. Step-scoped goal

```xml
<Step id="Step_05" name="Verify egg-info resolution mechanics">
  <Goal id="Goal_sub_01" scope="step" priority="optional">
    "Confirm the specific git-internal mechanism that resolved
    modify/delete cleanly — required for the runbook's afterword
    but not for the readiness verdict."
  </Goal>
  ...
</Step>
```

Scope-limited. The trace's main `Goal_01` is about merge readiness; this sub-goal is tangential, correctly marked `optional`.

## 5. What this enables that v0.1 can't

| Capability | v0.1 | v0.2 with `<Goal>` |
| --- | --- | --- |
| Monitor scores "did Subject pursue the task?" | No — no mechanical representation of the task | Yes — compare Findings against Goal.success_criteria |
| Trace says "I was trying to do X" to downstream agents | Only in prose, non-extractable | First-class, parseable |
| Regulator-readable "purpose of this AI action" | Inferred from Thinking blocks | Declared explicitly |
| Deadline-awareness (met vs met_late) | No | Yes via `deadline` field |
| Multi-goal task decomposition | Entire trace is one implicit goal | Multiple Goals with sub-scopes |

## 6. Alternatives considered

1. **Extend `<Constraint>` to carry goal-like semantics.** Rejected: Constraint binds `<Action>` atoms (limits on what can be done). Goal binds `<Finding>` atoms (what must be achieved). Different verification obligations; different atom.
2. **Attribute on `<Step>`.** Rejected: would conflate step purpose with trace purpose. A trace has one overarching goal; each step serves that goal. Keeping them as distinct atoms preserves that structure.
3. **Defer to v0.3.** Rejected: GCN experiment's I/O conformance metric *depends* on this atom. Deferring blocks the experiment.
4. **Introduce `<Objective>` instead of `<Goal>`.** Purely stylistic; Goal is the more common term in agent/RL literature. Pick and move on.

## 7. Dogfooding findings from writing this proposal

Writing the motivation and the proposed signature in Scholia surfaced several v0.1 ergonomics observations worth logging:

1. **Inline operators feel right.** `REFER:`, `IMPLIES:`, `LIST:` inside `<Thinking>` content read naturally and parse unambiguously.
2. **`<Deciding>` + `<Branch>` + `<Finding>` is coherent but verbose for simple two-option choices.** Consider v0.2 shorthand: `<Deciding options="A, B" chose="A">rationale</Deciding>` as sugar over the full form.
3. **No good place to record alternatives considered + rejected.** Section 6 above ("Alternatives considered") has no home in a v0.1 trace except as prose inside `<Deciding>`. Worth considering `<Alternative>` atom in v0.3+ — or just tolerating prose here.
4. **Cross-trace references.** This doc points at `.ralph/plans/scholia_gcn_experiment_plan.md` §6b. v0.1's cross-trace reference notation (`trace_id:atom_id`) doesn't naturally handle markdown documents that aren't Scholia traces. v0.3 candidate: generalize references to URIs.
5. **Category assignment was easy.** The six-category structure makes "where does this atom go?" a quick judgment. That's a good sign the taxonomy is load-bearing.

## 8. Open questions

1. **Should `<Goal>` be required in non-research traces?** Current proposal allows explicit research-mode opt-out. An alternative is requiring Goal unconditionally — harsher but cleaner. Needs operator input.
2. **Interaction with `<Retract>`.** If a trace retracts its own Goal mid-run (agent realizes the task was misspecified), what's the semantics? Probably: new Goal atom with `supersedes="Goal_01"` and a `<Retract>` of the original.
3. **Machine-readable success criteria.** Should `success_criteria` be restricted to a DSL (e.g., FOL fragments) so Monitors can mechanically evaluate them? Or freeform prose interpreted by an LLM Monitor? This maps directly onto the DFL NL→FOL adjudication path.

## 9. Acceptance recommendation

**Accept for Scholia v0.2** with the caveats noted in §8. Add to NOTATION_REFERENCE.md as atom 23 (in the meta category), update the atom-count in the preamble, add validity rule 8, ship v0.2 reference doc.
