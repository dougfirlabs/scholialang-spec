# Scholia v0.2 extension proposal - temporal instrumentation

**Status:** accepted for v0.2
**Target version:** Scholia v0.2
**Impact:** adds `timestamp` to `<Observation>` and `<Action>`, adds one meta atom.

## 1. Motivation

Scholia traces need to line up with the Events stream shown in a host application's GUI. v0.1 can say what an agent observed or did, but cannot point at the live event line that produced the observation or action.

## 2. Proposed attributes

`timestamp` is accepted only on `<Observation>` and `<Action>`.

```xml
<Observation id="Obs_01" timestamp="2026-04-20T23:03:48-07:00">
  command output observed here
</Observation>
```

The value is ISO-8601. It is optional and does not change v0.1 traces.

## 3. Proposed atom

```xml
<EventRef
  id="EventRef_01"
  instance="ot_A"
  run_id="rsi-20260420-230348"
  sequence="42"
  for="Obs_01"
  wall_clock="2026-04-20T23:03:48-07:00" />
```

`<EventRef/>` is a sidecar meta atom. It links a Scholia atom id to a concrete Events-stream row. `for` may reference any atom id or step id. `sequence` is parsed as an integer.

## 4. Backward compatibility

Traces without timestamps or EventRefs remain valid. Timestamps are deliberately not universal: interior reasoning atoms do not anchor directly to event-stream lines.

## 5. Alternatives considered

1. Universal `timestamp` attribute. Rejected because it implies reasoning atoms directly correspond to Events rows.
2. Store Events references in prose. Rejected because the GUI and Monitor need structured links.

## 6. Implementation estimate

Parser and dataclass changes are small. The only new parser behavior is attribute-level validation: `timestamp` must reject on atoms other than Observation and Action, and ISO values must be checked before constructing the AST.
