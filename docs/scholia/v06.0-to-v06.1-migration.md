# Scholia v0.6.0 → v0.6.1 migration note

**Audience:** producers (emitters) of Scholia traces, consumers
(validators, dashboards, enrichers), maintainers of runtime mirrors such
as `scholialang-mcp`, and authors of the `scholialang.org` spec page.

**Target reading time:** ~4 minutes.

**Source of truth:** [`SCHOLIA_v0.6_SPEC.md`](SCHOLIA_v0.6_SPEC.md) §10.10
(the v0.6.1 reconciliation change manifest), the `<Concluding>` attribute
table in [`../../reference/notation-reference.md`](../../reference/notation-reference.md),
and the published `scholialang` reference implementation.

---

## §0 TL;DR — additive, non-breaking

**v0.6.1 is additive and non-breaking. v0.6.0 traces and tooling stay
valid with zero changes; nothing is deprecated.** It contains exactly
one normative addition and one reconciliation:

1. **Optional `status` on `<Concluding>`** — `met` / `unmet` /
   `partially_met`. A status-less `<Concluding>` remains valid.
2. **Rule-8 (`goal_declared`) reconciliation** — the validator rule and
   the `<Concluding>` attribute table now agree that `status` is an
   allowed attribute, resolving the v0.6.0 contradiction.

No atom-kind changes (the closed set stays at 32). No other normative
edits. v0.6.0 is **not** deprecated.

This satisfies the Apache-2.0 §4(b) / CC-BY "state/indicate changes"
obligation for the v0.6.0 → v0.6.1 delta.

## §1 What changed

### §1.1 Optional `status` on `<Concluding>`

`<Concluding>` gains an optional `status` attribute with the enum
`met | unmet | partially_met`. It is the **completion verdict** on the
goal-close, and is deliberately distinct from the two attributes it sits
beside:

| Attribute | Answers | Enum / type |
|---|---|---|
| `status` (NEW) | *Did the goal complete?* | `met` / `unmet` / `partially_met` |
| `confidence` | *How certain is the close?* | `float [0,1]` |
| `criticality` | *How load-bearing is the close?* | `incidental … kernel` |

`status` is **optional**. A `<Concluding>` with no `status` stays valid
under every rule. It becomes *required* only as the means to satisfy rule
8 (`goal_declared`) when closing a `priority="required"` Goal — see §1.2.

```xml
<!-- v0.6.1: a required Goal closed directly by a status-bearing Concluding -->
<Concluding id="C_01" for_goal="G_01" status="met" confidence="0.85" criticality="ledger">
  REFER:F_01 AND REFER:Obs_01 IMPLIES the migration is complete.
</Concluding>
```

### §1.2 Rule-8 (`goal_declared`) reconciliation

In v0.6.0 the spec was internally inconsistent: validator rule 8 expected
a `status` on the closing `<Concluding>`, but the `<Concluding>`
attribute table omitted `status` (it listed only `for_goal` /
`confidence` / `criticality`), and the reference parser enforced that
omission — so a producer that followed rule 8 literally emitted a
`<Concluding status="met">` the reference parser **rejected**.

v0.6.1 reconciles the two surfaces: `status` is now a first-class
optional attribute on `<Concluding>` (§1.1), so rule 8 and the attribute
table agree. Rule 8 now reads: a `priority="required"` `<Goal>` is closed
by a `<Concluding for_goal="…" status="…">` whose `status` is one of
`met` / `unmet` / `partially_met` (or, for back-compat, a
`<Finding for_goal="…" status="…">`, whose own table additionally carries
the legacy `met_late`). `<Meta:research-mode/>` still exempts the trace.

## §2 Producer guidance (MAY set status on Concluding)

- **You MAY set `status` on a `<Concluding>`.** Prefer it when the
  Concluding closes a `priority="required"` Goal — it is the direct,
  rule-8-satisfying form, and it is now legible to the attribute table
  and the (v0.6.1) reference parser.
- **You are not required to.** A status-less `<Concluding>` is still
  valid. Existing emitters need no change.
- **`status` ≠ `confidence` ≠ `criticality`.** Use `status` for the
  completion verdict, not for certainty or load-bearing rank.
- **Back-compat close path unchanged.** Closing a required Goal with a
  `<Finding for_goal="…" status="…">` still works (the `for_goal` alias
  remains deprecated, removal deferred to v0.7; see
  [`v05-to-v06-migration.md`](v05-to-v06-migration.md) §1.6).

## §3 Consumer guidance (treat status as optional)

- **Treat `<Concluding status>` as optional.** Do not require it; a
  status-less close is valid. Do not hard-fail on its presence.
- **Status-less closes stay valid.** No consumer that accepted v0.6.0
  traces should reject a v0.6.0 trace under v0.6.1.
- **Version stamp.** The validator report's
  `scholia_validator_version` reads `0.6.1` on the reconciliation build
  and `0.6.0` on the pre-reconciliation build; the report **shape** is
  identical. Do not hard-code a rule-name or version allowlist.

## §4 Downstream propagation checklist

v0.6.1 is the spec-repo change. The following surfaces consume the spec
and must be propagated in dependency order (each is a separate,
spec-gated change — not part of this spec repo):

- [ ] **`scholialang` reference implementation** — add the optional
      `status` field to the `Concluding` dataclass and to its closed-set
      wire attributes (`KIND_SPECIFIC_FIELDS["Concluding"]`), so
      `<Concluding status="met">` parses and validates; bump
      `SCHOLIA_VALIDATOR_VERSION` to `0.6.1`. (Until this lands, the
      published build stamps `0.6.0` and rejects `status` on
      `<Concluding>` — the v0.6.1 golden record in
      `compatibility-manifest.json` documents the target contract.)
- [ ] **`scholialang-mcp` vendored copies + dependency pin** — re-vendor
      the updated atom/validator surfaces and bump the `scholialang`
      dependency pin to the v0.6.1 build.
- [ ] **`scholialang.org` spec page** — refresh the rendered spec and the
      `<Concluding>` attribute table to show optional `status`, and the
      reconciled rule 8.
